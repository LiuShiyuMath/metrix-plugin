1 · 鸭鸭你好

CEO 鸭鸭，先深呼吸。这本册子不会让你看见任何 npm install、不会让你看见任何花括号。你只需要看故事、看流程、看几张图就行。

一句话版本：
我们做了一个叫 TeamAgent 的小工具，能让 Claude Code（员工 A 在用的那个 AI 编程助手）记住人犯过的错，下一次员工 B 同样的错就被自动拦住，然后给你（CEO）一份证据报告，证明这次没白拦。

简单说：

  • 员工教过 AI 一次的话，AI 就记住了。
  • 下次另一个员工碰到 AI，AI 不会再交同一份错的作业。
  • 你随时能看证据，知道这次拦住的是真实的重复错误，而不是工具自己吹牛。

这事情值钱在哪里？小公司一个月里有 30% 的 PR 时间，是在重复纠正 AI 之前已经被纠正过的同类错误。TeamAgent 把这 30% 直接砍掉。

2 · 为什么要做这个

鸭鸭你想象一下这个场景：

  +-----------+        "用 dayjs，不要用 moment！"
  | 员工 Alice |  -------------------------------+
  +-----------+                                   |
        |  说完气得喝了口水                        |
        v                                          v
  +------------+                            +------------+
  | Claude     |  "好的好的，下次记住"        | Claude (3天后) |
  | (周一)     |                            | (周四，新会话)|
  +------------+                            +------------+
        |                                          |
        |                                          v
        |                                  "好的我用 moment.js"
        |                                          |
        v                                          v
                                         +------------+
                                         | 员工 Bob   |  "??????"
                                         +------------+

问题在于：Claude Code 每开一个新会话，它就把记忆清空了。Alice 教过它一次「不要用 moment」，三天后 Bob 开新会话，AI 又交了同一份错作业。

CEO 视角：这就是一笔看不见的浪费。错误本身不大，但 3 个员工 × 每周 5 次 × 每次 15 分钟来回，一个月就是 15 小时白送。

TeamAgent 干的事很朴素，就 3 件：

  1. 抓住 Alice 的纠正 — Alice 说「不要用 X，用 Y」的时候，悄悄存下来。
  2. 拦住 Bob 的重复 — Bob 三天后让 Claude 装 moment，TeamAgent 在 Claude 真的去装之前喊停，把 Alice 的原话翻出来给 Bob 看。
  3. 把证据汇报给你 — 自动生成一张「上次 Alice 教过这个 / 这次 Bob 差点踩坑 / 我拦住了」的网页，给你一目了然。

5 · 记忆鸭 · teamagent-memory

这是三只鸭里最重要的那一只。其它两只都是围着它转的。

5.1 它在什么时候动手？

Claude Code 在工作的时候，会经过几个固定的「门」。记忆鸭就守在三道门后：

门什么时候触发记忆鸭干什么
Stop 门Claude 给完一轮回答，准备闭嘴时翻最近 40 条聊天记录，找有没有「不要 X，用 Y」这种纠正话；找到了就存成规则卡片
PreToolUse 门Claude 想真的去敲 Bash 命令时（装包、改文件之类）对着规则卡片库扫一眼，命中了就喊停，把卡片里的原文还给 Claude 看
UserPromptSubmit 门用户刚按下回车，但 Claude 还没开口看用户句子里有没有以前犯过的关键词；有，就悄悄给 Claude 塞个备忘：「兄弟，上次这件事你被骂过」

5.2 一张规则卡片长什么样？

{
  "id":         "rule-2026-05-13-moment-dayjs",
  "trigger":    { "tool": "Bash", "pattern": "npm install moment" },
  "wrong":      "Adopting moment.js as a new dependency",
  "correct":    "Use dayjs",
  "why":        "moment is in maintenance mode",
  "confidence": 1,
  "captured_at":"2026-05-13T00:00:00Z"
}

简单说，每张卡片 = 一句教训。它带：

  • 什么动作会触发（trigger）
  • 错在哪里（wrong）
  • 正确应该是什么（correct）
  • 为什么这样（why）
  • 信心度（confidence，被命中过几次）
鸭鸭说：你可以把规则卡片想成「钉在墙上的便签纸」。墙是公司，便签是规矩。AI 每动一次手，都先抬头看看墙。

5.3 它不会犯什么错？

  • 不会乱拦：只有 Bash 命令真的命中卡片里的 pattern 才会拦。装 dayjs 不会被拦，装别的也不会。
  • 不会重复存：Alice 教了一次，再教一次同样的话，卡片不会变成两张，只会把 confidence 从 1 加到 2。
  • 不会卡死会话：如果记忆鸭自己出问题（比如卡片文件乱了），它就默默退出，不会让 Claude 用不了。Fail open 不是 fail close。

6 · 证据鸭 · teamagent-proof-console

记忆鸭只管抓和拦。证据鸭只管把这事翻译给 CEO 看。

6.1 它产出什么？

三个东西，都在 evidence/ 这个文件夹里：

文件给谁看说明
rule-card.json合规 / 审计那张被命中的规则卡片原件
ceo-summary.htmlCEO 鸭鸭一页网页，含八步故事 + 规则卡片 + before/after 对比
ceo-demo.mp4大会演讲不超过 90 秒、不低于 720p 的演示录屏（可选）

6.2 那张 CEO 网页里必须含什么？

有四句话是硬合同，缺一不可（裁判鸭会抓）：

  1. 「Previous Claude Code made this mistake」（上一只 Claude 犯过这个错）
  2. 「TeamAgent blocked it」（TeamAgent 把它拦下来了）
  3. 「rule-card」（这就是那张卡片）
  4. 「before/after」（前后对比）

而且网页大小至少 2KB，确保不是空壳。

6.3 一条命令就生成

用户（或者你 CEO 鸭鸭自己）只要在 Claude Code 里敲：

/teamagent-proof-console:proof

证据鸭就会按顺序：

  1. 挑出最近一次被拦住的事件
  2. 把那张规则卡片渲染成 HTML
  3. 核对四锚点是否齐全
  4. 把结果写到 evidence/
这一步不会编故事。如果记忆鸭里没真发生过的事，证据鸭会拒绝生成，而不是凭空捏造一个好看的报告。CEO 鸭鸭看见的每张证据，背后都有真实的 hook 事件。

7 · 同步鸭 · teamagent-team-sync

7.1 它要解决什么问题？

Alice 在她那台 Mac 上教 Claude「不要用 moment」，记忆鸭把卡片存在 Alice 的电脑里。但 Bob 用的是他自己的电脑，他电脑里没有这张卡片，记忆鸭看不见。这是单兵抓人，没有团队记忆。

同步鸭就是来当那个「公司共享文件柜」的：

  Alice 电脑                  团队共享文件柜               Bob 电脑
  +----------+      推       +----------+      拉       +----------+
  | 记忆鸭 A |  ---------->  | team/    |  ---------->  | 记忆鸭 B |
  | 卡片库   |               | rules    |               | 卡片库   |
  +----------+               | .jsonl   |               +----------+
                             +----------+

7.2 它什么时候动手？

时机做什么
新会话开始（SessionStart）从团队文件柜拉新卡片；本地没有的就拷过来
用户说要分享（UserPromptSubmit）识别到「publish this rule / 分享给团队」这类话，提醒 Claude 调用发布技能

7.3 两人规则不一样怎么办？

Alice 说「不要用 moment，用 dayjs」。
Bob 说「不要用 moment，用 date-fns」。
两张卡片触发条件一样，正确答案不同。同步鸭不会替你做决定，它把冲突写到一个叫 conflicts.jsonl 的小本子里，下次会话开始时给 Claude 一个提醒：

「兄弟，团队里这条规矩有两个版本，你跑之前先问问人类决定要哪个。」

裁决权回到人手里。鸭鸭可以选：

  1. 保留本地（Alice 的版本不变）
  2. 接受团队（用 Bob 的版本覆盖）
  3. 合并（两个都不要 moment，候选答案 dayjs/date-fns）
  4. 两个都丢（这条规矩本来就不该存在）
鸭鸭说：这个设计来自一条铁律 —— 分布式系统里同步可以自动，决策必须人来。同步鸭做搬运工，不做法官。

8 · 裁判鸭 · 第三方机械裁判

8.1 为什么要有它？

CEO 灵魂提问：「你说这工具好用，但你说了不算 —— 你怎么证明？」

这是 TeamAgent 设计里最重要的一节。CEO 鸭鸭你听好：

不要让代码自己评价自己。
任何「我们做得多好」的话，都不能由做这件事的代码自己说。要请一个不沾边的第三方看证据。

这一原则在我们的体系里长成了一只裁判鸭。它的工作流程是：

         记忆鸭           证据鸭           同步鸭
            \              |              /
             \             |             /
              \            v            /
               +-------------------------+
               |   裁判鸭 (judge.sh)     |
               | 1. 跑三个固定测试探针    |
               | 2. 把结果写成 judge.json|
               | 3. 不做总结，只做事实   |
               +-------------------------+
                          |
                          v
               +-------------------------+
               | 另一只小 LLM (空白脑)   |
               | 只读 judge.json        |
               | 输出 PASS / FAIL       |
               +-------------------------+
                          |
                          v
                    CEO 鸭鸭看结果

8.2 三只探针测的是什么？

探针验什么怎么验
stream-json Claude 本身真在产事件流，证据不是伪造的 跑一次 claudefast --output-format stream-json -p "ping"，看是否有 JSON 事件输出
ab-plugin-dir 记忆鸭真的有因果效应：装了它会拦，不装就不拦 跑三轨：
A 不装鸭子 + 坏指令 → 不该拦
B 装鸭子 + 坏指令 → 应该拦
C 装鸭子 + 良性指令 → 不该误伤
file-checks CEO 报告四锚点齐 / 卡片 schema 合规 / 代码语法对 / git 已提交 用 jq、grep、wc -c、node --check、git status 五件套

8.3 裁判鸭也不能自己说赢

裁判鸭把所有事实写到 judge.json 后，不下结论。结论交给一个另一只开着空白脑（--plugin-dir /tmp/empty）的小 LLM，让它只能看 judge.json 上的字段，回答 PASS 或 FAIL。

这只「空白脑 LLM」连记忆鸭、证据鸭、同步鸭都看不见 —— 它没办法被忽悠。

鸭鸭说：这就像审计师不能用自家公司开的 office 来写审计报告。证据要留痕，工具要无关，结论要外人下。

9 · 怎么把鸭子装进 Claude Code

不用写代码。三条命令搞定。

9.1 让 Claude Code 知道我们这个市场

/plugin marketplace add ./teamagent-marketplace

（如果是远端 GitHub 仓库，就改成 git@github.com:libz-renlab-ai/TeamBrain。）

9.2 装其中一只鸭子（按需要）

/plugin install teamagent-memory@teamagent-marketplace
/plugin install teamagent-proof-console@teamagent-marketplace
/plugin install teamagent-team-sync@teamagent-marketplace

三只都装也行，只装记忆鸭也行。一只都不装也行（但那样就没东西了）。

9.3 让裁判鸭跑一次自检

cd teamagent-marketplace
bash bin/judge.sh
bash bin/judge-verdict.sh

看见两个 PASS 就行。

9.4 给 CEO 鸭鸭看的演示

开两个 Claude Code 窗口：

  1. 第一个窗口：装鸭子，让 Alice 角色跟 Claude 说「不要用 moment，用 dayjs」。然后看 ~/.teamagent/rules.jsonl，会出现一张卡片。
  2. 关第一个窗口，开第二个：装鸭子，让 Bob 角色跟 Claude 说「装一下 moment」。Claude 在最后一秒会喊停，并把 Alice 那句话翻出来。
  3. 再敲 /teamagent-proof-console:proof，那张 CEO 网页就生成在 evidence/ceo-summary.html。
鸭鸭说：这套演示三分钟内可以走完，视频录屏不超过 90 秒。这就是裁判鸭里 ffprobe 那一条的来源 —— 我们要求 CEO 演示越短越好。

10 · CEO 灵魂三问

问题一：这玩意儿会不会拦错好东西？

会，理论上。但我们做了三层防护：

  1. 规则只匹配 Bash 命令，不会拦 Read / Write / Edit 之类的常规操作。
  2. 规则的 trigger pattern 有长度上限 512 字符，防止有人塞一条「拦一切」的恶意正则进来。
  3. 裁判鸭的 C 轨专门测「良性指令在装了鸭子的情况下是不是还能跑」。装新规则前，C 轨必须依然过。

问题二：如果规则错了 / 过时了，怎么办？

三条路：

  • 用 teamagent 命令行的 clear 子命令清空（有 --yes 二次确认）。
  • 手工删 ~/.teamagent/rules.jsonl 里那一行。
  • 或者下一次会话直接跟 Claude 说「moment 现在可以用了」，让记忆鸭再抓一次「相反」的纠正，覆盖前一张卡片。

问题三：商业模式是什么？

三种可能（CEO 鸭鸭你拍板）：

模式对谁卖核心价值
免费 + 开源个人 / 小团队引流到团队版
团队订阅5 - 50 人开发团队同步鸭的中央服务 + CEO Dashboard
企业版大公司 / 合规要求高审计轨迹、SAML、自托管裁判鸭
鸭鸭压轴说：这三只鸭子加一只裁判，本质上做的是「把 AI 团队的口头规矩，变成可执行、可证明、可同步的卡片」。把口头规矩固化成机械合同，这是过去 5 年所有 AI 工程提效工具里，最被低估的那条线。
